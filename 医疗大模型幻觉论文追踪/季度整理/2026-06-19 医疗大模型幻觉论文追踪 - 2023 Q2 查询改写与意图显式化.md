# 医疗大模型幻觉论文追踪 - 2023 Q2 查询改写与意图显式化

本次检索时间桶：2023 Q2（2023-04-01 到 2023-06-30）

## 检索范围说明

- 本次严格只纳入在 `2023-04-01` 到 `2023-06-30` 之间首次发表、公开或可检索到的相关论文。
- 主题限定为：通过意图识别、澄清式交互、查询路由、回答前判别、query rewriting、intent clarification、under-specification handling 等方式治理 hallucination。
- 本季度检索到的结果有一个明显特点：
  - 直接写“医疗 RAG + ask/rewrite/conservative 路由”的公开论文仍然很少。
  - 这一桶里更早成形的是两条基础技术线：`query rewriting for RAG` 与 `machine intent explicitization / interactive query understanding`。
  - 医疗侧在这一季度更像是“安全医疗 QA 能力基线”的建立，而不是上游输入治理框架的成熟论文。
- 因此本次结果分为两类：
  - 直接相关：明确讨论 query rewriting、query understanding、under-specified query 处理。
  - 医疗参照：虽然不是 ask/rewrite 路由论文，但能为“医疗幻觉治理为什么需要前置输入控制层”提供场景依据。

## 一、论文汇总

### 1. Query Rewriting for Retrieval-Augmented Large Language Models

- 作者：Xinbei Ma, Yeyun Gong, Pengcheng He, Hai Zhao, Nan Duan
- 首次公开：2023-05-23
- 链接：[arXiv:2305.14283](https://arxiv.org/abs/2305.14283)
- 摘要：
  - 论文提出把传统 `retrieve-then-read` 改成 `rewrite-retrieve-read`，核心观点是：RAG 的瓶颈不只在检索器和阅读器，还在“原始用户输入并不等于最适合检索的查询”。
  - 作者先用 LLM 生成查询改写，再调用搜索引擎取回上下文；进一步又用一个小型可训练 rewriter，在冻结 reader 的条件下通过强化学习学习更适合下游回答的查询。
- 主要贡献：
  - 把 `query rewriting` 从辅助技巧提升成 RAG 框架中的一级组件。
  - 提出 `LLM rewriter` 与 `trainable rewriter` 两种路线，并用 reader 反馈做强化学习对齐。
  - 在 `HotpotQA`、`AmbigNQ`、`PopQA` 以及多项选择 QA 设置上评估，指标包括 `EM`、`F1` 和 retrieval `Hit`，整体显示改写能稳定提升下游问答表现。
- 为什么和你的题目有关：
  - 你现在的 `rewrite` 分支可以直接借用它的核心论点：患者原始问句不是检索器最优输入，先做检索友好化改写是合理且必要的。
  - `AmbigNQ` 结果尤其重要，因为它说明“含歧义的问题”对 rewrite 更敏感，这与你的 `多意图 / 信息不足 / 需要追问` 主线高度一致。

### 2. Query Understanding in the Age of Large Language Models

- 作者：Avishek Anand, Venktesh V, Abhijit Anand, Vinay Setty
- 首次公开：2023-06-28
- 链接：[arXiv:2306.16004](https://arxiv.org/abs/2306.16004)
- 摘要：
  - 这是一篇 position paper，但和你的课题非常贴近。作者主张：LLM 时代的 query understanding 不应只输出一个隐藏的系统内部意图，而应把 `machine intent` 用自然语言显式写出来，让用户在最终检索或回答前先 `commit / edit / explore`。
  - 论文把 under-specified query 的处理从“后台自动纠偏”改成“前台可交互、可编辑、可澄清”的流程。
- 主要贡献：
  - 提出 `interactive query rewriting` 框架，把 machine intent 显式化，支持 `soft rewrite`、`hard rewrite`、探索与 refinement。
  - 明确把 query understanding 拆成多个可研究问题：如何最小化 hard rewrite、如何保证 machine intent faithful、如何在多种可能意图之间做多样化展示。
  - 论文使用 `TREC Web` topic 和 `Google People Also Ask` 数据构造初步实验，强调比传统 query classifier / spell checker / expansion 更透明的人机协同式意图理解。
- 为什么和你的题目有关：
  - 你想做的 `ask / rewrite / direct / conservative`，本质上就是在回答前显式暴露并修正机器对用户意图的理解。
  - 这篇论文可以直接支撑你把“是否追问”和“追问什么 / 改写成什么”建模为一个上游交互式意图显式化问题，而不是只靠一个黑盒分数。

### 3. Towards Expert-Level Medical Question Answering with Large Language Models

- 作者：Karan Singhal, Tao Tu, Juraj Gottweis, Rory Sayres 等
- 首次公开：2023-05-16
- 链接：[arXiv:2305.09617](https://arxiv.org/abs/2305.09617)
- 摘要：
  - 论文提出 `Med-PaLM 2`，目标是提升医疗问答中的准确性、临床效用与安全性。
  - 作者在 `MedQA`、`MedMCQA`、`PubMedQA`、`MMLU clinical topics` 等基准上评测，并新增 `240` 条 adversarial long-form questions，用医生和普通用户从 factuality、clinical utility、harm likelihood 等角度做人评。
- 主要贡献：
  - 给出了 `2023 Q2` 医疗问答能力与安全评估的重要强基线。
  - 强调医疗问答不能只看自动 benchmark，还要看 `possible harm extent`、`possible harm likelihood`、是否遗漏关键信息、是否符合医学共识。
  - 用 `MultiMedQA 1066` 和 adversarial sets 说明：即便模型性能显著提升，医疗场景仍然需要更细致的安全控制。
- 为什么和你的题目有关：
  - 它不是 query clarification 论文，但能作为你论文里的医疗侧动机证据：医疗场景对 factuality 与 harm 极其敏感，因此在回答前增加输入质量控制与策略路由是有现实必要性的。
  - 你后续做 `conservative answer`、`高风险输入优先追问` 时，可以直接参考它的安全评价维度，而不只看正确率。

## 二、与《硕士论文进展》的关系

你的当前主线是：

- 识别患者问题中的多个医学意图。
- 判断输入是否清晰、医学相关、适合直接检索。
- 根据输入质量决定：直接 RAG、先改写、还是追问/保守回答。

这三篇论文和你当前设计的关系可以压缩成下面这张映射：

| 你的模块 | 对应论文 | 可借用点 |
| --- | --- | --- |
| rewrite 前置层 | Query Rewriting for Retrieval-Augmented Large Language Models | 原始问句不等于检索最优输入；可单独训练 rewriter；用下游 QA 反馈反向优化 rewrite |
| ask / intent explicitization | Query Understanding in the Age of Large Language Models | 先把 machine intent 写出来，再让用户 commit / edit / explore；适合 under-specified query |
| 医疗安全评估与 conservative | Towards Expert-Level Medical Question Answering with Large Language Models | 医疗问答必须看 harm、factuality、completeness，不应只看自动准确率 |

如果把你的论文问题再明确一点，可以写成：

> 在医疗 RAG 之前增加一个输入治理层，对患者问题执行多意图识别、输入质量评估与策略路由，决定是直接检索、先做检索友好化 rewrite、先追问澄清，还是触发保守回答，从而降低医疗大模型幻觉与潜在风险。

从这个角度看：

- `Query Rewriting for RAG` 说明 `rewrite` 是核心变量，不是附属模块。
- `Query Understanding...` 说明“意图显式化 + 可交互修正”是合理方向，尤其适合欠明确输入。
- `Med-PaLM 2` 说明医疗侧评估必须把 harm 和临床可用性单独拿出来。

## 三、我可以直接学习的思路

### 1. 方法设计思路

- 不要把 `rewrite` 只理解成语言润色；它更应该是 `retrieval-oriented rewrite`。
- 可以把 `ask` 路由解释成 `machine intent 还不够 faithful，不能直接 commit 到 retrieval / answer`。
- 你的系统可以显式拆成四路：
  - `direct`
  - `rewrite`
  - `ask`
  - `conservative`
- 其中 `rewrite` 再细分成两类：
  - `disambiguating rewrite`：把含糊问法变成更单一、更可检索的表达。
  - `retrieval expansion rewrite`：补足症状、部位、时间、对象等检索友好线索。

### 2. 实验方法

- 做两段式验证：
  - 第一段只测 `输入治理层` 是否把原始 query 变得更适合检索。
  - 第二段把治理层接到医疗 RAG 上，看最终回答质量是否提升。
- `rewrite` 支线可做：
  - 无 rewrite
  - 直接 LLM rewrite
  - 基于规则/模板的轻量 rewrite
  - 用下游 reward 优化的 rewrite
- `ask` 支线可做：
  - 不追问
  - 低阈值追问
  - 高阈值追问
  - oracle 追问
- `route` 支线可做：
  - 二分类：`direct` vs `non-direct`
  - 四分类：`direct / rewrite / ask / conservative`
  - 观察哪些错误主要来自 `under-specification`，哪些来自 `多意图冲突`

### 3. 数据集思路

- 继续用你现有 `MedDialog-CN` 患者问题作为主语料。
- 专门构造一个 `under-specified medical query` 子集，重点包含：
  - 症状有了，但缺部位/时长/对象。
  - 一个输入里混有多个医学意图。
  - 口语化表达导致检索词不稳定。
  - 需要先澄清才能安全回答的问题。
- 可以借鉴 `AmbigNQ` 的思路，给每个原始问句补：
  - 是否歧义
  - 可能的多个 intent
  - 理想澄清问题
  - 澄清后可提交给 RAG 的标准化 query

### 4. 评测指标

- 多意图识别：
  - `micro-F1`
  - `macro-F1`
  - `hamming loss`
- rewrite：
  - `Recall@k`
  - `MRR`
  - `nDCG`
  - top1 evidence hit
  - rewrite 后 route 改善率
- ask：
  - `need_ask` accuracy / F1
  - 追问后 retrieval hit rate 提升
  - 平均追问轮次
- 最终医疗 QA：
  - factuality
  - completeness
  - possible harm extent
  - possible harm likelihood
  - expert pairwise preference

## 四、可直接引用到论文里的点

### 1. 动机层

- 幻觉治理不应只在回答后做核验，也应在回答前做输入治理。
- 医疗问答里的高风险错误，很多源头不是“模型不会答”，而是“用户输入本身欠明确、欠完整、欠规范”。

### 2. 方法层

- `Query Rewriting for Retrieval-Augmented Large Language Models` 可支撑：
  - 原始 query 与最优检索 query 之间存在系统性差距。
  - rewrite 应直接为下游 retrieval / QA 服务，而不是只追求表面文本自然。
- `Query Understanding in the Age of Large Language Models` 可支撑：
  - 机器对用户意图的解释应显式化、可编辑、可澄清。
  - 对 under-specified query，先暴露 machine intent 再检索，比直接答复更安全。
- `Med-PaLM 2` 可支撑：
  - 医疗场景需要单独衡量 harm 与 clinical utility。
  - 你的 `conservative` 路由有充分场景合理性。

### 3. 你这条线的差异化

- 不是只做医疗 QA。
- 不是只做医疗 RAG。
- 不是只做 rewrite。
- 而是把 `多意图识别 + 输入质量判别 + 交互式澄清/改写 + 医疗安全路由` 统一成一个医疗 RAG 上游控制层。

## 五、这次时间桶的结论

- `2023 Q2` 这一桶里，还看不到成熟的“医疗 RAG + ask/rewrite/conservative 路由”一体化论文。
- 但已经很清楚地出现了三条后来会汇合的线：
  - `RAG 查询改写`：Query Rewriting for Retrieval-Augmented Large Language Models
  - `意图显式化与交互式 query understanding`：Query Understanding in the Age of Large Language Models
  - `医疗问答安全评估`：Med-PaLM 2
- 对你的硕士论文来说，这一桶最重要的价值不是找到现成同题论文，而是证明你的问题定义正处在几个关键方向的交叉点上。

  - 后续更直接命中“ambiguous question + RAG clarification”的代表作，例如 `Tree of Clarifications`，首次公开时间已是 `2023-10-23`，不属于本桶。
