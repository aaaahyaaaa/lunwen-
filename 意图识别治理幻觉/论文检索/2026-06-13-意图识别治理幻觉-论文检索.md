## 论文 1

### RAGRouter-Bench: A Dataset and Benchmark for Adaptive RAG Routing

- 作者：Ziqi Wang, Xi Zhu, Shuhang Lin, Haochen Xue, Minghao Guo, Yongfeng Zhang
- 时间：arXiv 2026-01-30
- 链接：[arXiv](https://arxiv.org/abs/2602.00296)

#### 摘要

这篇论文把 **adaptive RAG routing** 正式做成了 benchmark。作者指出，已有 RAG 工作大多只比较某一种检索范式是否更强，却缺少一个统一框架来回答：**什么样的 query 应该走什么样的 RAG 范式**。因此论文构建了 `RAGRouter-Bench`，覆盖 7,727 个 query 和 21,460 个文档，统一评估不同 RAG 路径在生成质量与资源成本上的权衡。实验结论很重要：**没有一种 RAG 路径对所有 query 都最优**，真正决定效果的是 query 与 corpus 的匹配关系。

#### 主要贡献

1. 提出面向 adaptive RAG 的专门 benchmark，而不是只做单模型比较。
2. 把 RAG 路由问题从“经验调参”提升为“可训练、可评测”的正式任务。
3. 明确指出 query-corpus interaction 会影响最优路由，不存在万能的单一路径。

#### 和你论文的关系

这篇论文能直接支撑你把 `Q_route` 写得更像一个正式研究对象：

> `Q_route` 不是经验分数，而是一个 **query-aware routing policy**。

你现在做的是：

- 多意图识别
- 输入语义质量评分 `Q_sem`
- `direct / rewrite / ask / conservative` 路由

RAGRouter-Bench 的意义在于，它说明“**不同问题本来就应该走不同路径**”是合理且可评估的。你可以据此把自己的论文表达成：

> 对医疗患者问题，系统应先估计 query 的意图完整性与可答性，再决定是否直接进入 RAG，还是先 rewrite、先 ask、或直接 conservative。

#### 你可以学习的思路

- 把你的路由器定义成一个正式分类/策略任务，而不是仅作为工程 heuristics。
- 后续可以把 `Q_route` 的目标写成：
  - `direct`
  - `rewrite`
  - `ask`
  - `conservative`
- 你的路由标签天然比通用 RAGRouter-Bench 更细，也更适合医疗场景。

#### 可借鉴的实验方法

- 先不急着比较复杂生成模型，先把“**路由本身是否分对**”单独评测。
- 设计两阶段实验：
  - 阶段 1：route prediction 是否正确
  - 阶段 2：按 route 执行后，最终 hallucination 是否下降

#### 可借鉴的数据集 / 实验设计 / 指标

- 论文数据集：`RAGRouter-Bench`
- 你可以新增指标：
  - `route accuracy`
  - `route macro-F1`
  - `unsafe direct-answer rate`
  - `token-cost per successful case`

#### 可引用的地方

- 相关工作：支撑 RAG 路由是独立研究问题。
- 方法章节：说明你的 `Q_route` 属于 query-aware adaptive routing。

## 论文 2

### Lightweight Query Routing for Adaptive RAG: A Baseline Study on RAGRouter-Bench

- 作者：Prakhar Bansal, Shivangi Agarwal
- 时间：arXiv 2026-04-03
- 链接：[arXiv](https://arxiv.org/abs/2604.03455)

#### 摘要

这篇论文非常贴近你当前的技术路线。作者不走复杂 agent 或大模型决策，而是直接用 **TF-IDF、句向量、结构特征** 配合传统分类器做 query routing。结果显示，最好的配置 `TF-IDF + SVM` 在 `RAGRouter-Bench` 上达到 **macro-F1 0.928 / accuracy 93.2%**，同时可节省约 **28.1%** 的 token 开销。一个很关键的发现是：**表层词汇模式有时比语义向量更适合做路由判别**；而且论文特别指出，**medical queries 是最难路由的一类**。

#### 主要贡献

1. 给 adaptive RAG routing 提供了一个强而轻量的 classical baseline。
2. 证明 query routing 不一定要靠大模型推理，小模型/稀疏特征也可以很有效。
3. 指出医疗 query 的路由难度更高，这与你的研究场景直接相关。

#### 和你论文的关系

这篇论文和你现在的 baseline 风格几乎完全同路：

- 你现在用的是 `TF-IDF + Logistic Regression` 多标签意图分类。
- 这篇论文说明 **轻量级 query router 本身就是成立的研究路线**。

它对你的直接启发是：

> 你完全可以把 `Q_route` 或 route classifier 也先做成一个 CPU-friendly baseline，而不是一上来就用复杂 LLM policy。

#### 你可以学习的思路

- 把 route prediction 单独做成一个浅层分类任务。
- 先用 lexical feature 建一个强 baseline，再讨论是否引入更复杂特征。
- 意图分类输出可以反向喂给 route classifier，例如：
  - 预测出的意图数量
  - 最大/次大意图置信度差
  - 是否包含高风险诊断/用药类意图

#### 可借鉴的实验方法

- 做一个 `route baseline` 对比：
  - 规则阈值版 `Q_route`
  - `TF-IDF + LR/SVM`
  - `TF-IDF + route features + LR/SVM`
- 单独报告不同 route 类别的混淆情况，尤其看：
  - `ask` 被误判成 `direct`
  - `rewrite` 被误判成 `direct`

#### 可借鉴的数据集 / 实验设计 / 指标

- 可借鉴指标：
  - `macro-F1`
  - `per-route precision/recall`
  - `token saving`
  - `unsafe direct-answer false negative rate`

#### 可引用的地方

- 实验章节：支撑你选择小模型做路由 baseline 是合理的。
- 讨论章节：说明医疗 query 路由更难，因此你的场景具有研究价值。

## 论文 3

### Uncertainty-Aware Clarification in LLM Agents with Information Gain

- 作者：Mengyi Deng, Zhiwei Li, Xin Li, Tingyu Zhu, Ying Zhao, Zhijiang Guo, Wei Wang
- 时间：arXiv 2026-06-02
- 链接：[arXiv](https://arxiv.org/abs/2606.03135)

#### 摘要

这篇论文研究一个和你 `ask` 路由高度相关的问题：**什么时候值得先澄清，而不是直接继续回答/执行**。作者把澄清行为建模为对潜在用户目标不确定性的消解，并提出 **Information Gain Reward**，衡量某个 clarification question 是否真正缩小了目标空间。实验在 clarification-enhanced `τ-Bench` 上进行，结论是：基于信息增益优化的澄清策略，相比不澄清 baseline 能稳定提高任务成功率，而且额外交互成本很小。

#### 主要贡献

1. 把“该不该追问”从启发式规则推进到不确定性感知决策。
2. 用信息增益而不是表面自然度来定义好的 clarification。
3. 证明澄清问题可以做成独立策略模块，而不是 prompt 小技巧。

#### 和你论文的关系

这篇论文很适合用来升级你现在的 `ask` 路由表述：

> `ask` 不应只是“低于阈值就追问”，而应是“当问题存在关键不确定性，且追问的预期收益大于交互成本时才追问”。

这和你当前医疗论文场景特别契合，因为患者问题常见的风险正是：

- 多意图纠缠
- 关键槽位缺失
- 症状描述不完整
- 患者猜测与事实混杂

#### 你可以学习的思路

- 在 `Q_route` 中加入 `clarification worthiness` 的概念。
- 将 `ask` 与 `rewrite` 分开：  
  `rewrite` 解决表达杂乱，`ask` 解决信息缺失。
- 给低质量医疗输入人工标注：
  - `need-ask`
  - `ask-useful`
  - `ask-not-worth-it`

#### 可借鉴的实验方法

- 对同一批低质量患者问题，比较三种路由：
  - 固定阈值追问
  - 风险规则追问
  - 不确定性/信息增益驱动追问
- 单独统计：
  - 本该追问却直接回答的比例
  - 追问后是否补到了关键医学信息

#### 可借鉴的数据集 / 实验设计 / 指标

- 可借鉴指标：
  - `necessary-ask recall`
  - `unnecessary-ask rate`
  - `critical-slot hit rate`
  - `post-clarification answer gain`

#### 可引用的地方

- 方法章节：支撑 `ask` 是 uncertainty-aware policy。
- 实验章节：支撑你单独评测“追问触发是否合理”。

## 论文 4

### Do No Harm? Hallucination and Actor-Level Abuse in Web-Deployed Medical Large Language Models

- 作者：Sunday Oyinlola Ogundoyin, Muhammad Ikram, Rahat Masood
- 时间：arXiv 2026-05-20
- 链接：[arXiv](https://arxiv.org/abs/2605.20591)

#### 摘要

这篇论文不是专门做 intent recognition，但它对你的医疗论文非常重要，因为它把问题拉回到了**真实医疗部署风险**。作者大规模评测了 6,233 个在线医疗 GPT，抽样分析其中 1,500 个，并与 10 个开源模型比较，提出 `MedGPT-HEval` 等评测框架，从 factual accuracy、policy violation、privacy disclosure 等多个维度衡量风险。结果显示，相当比例的 web-deployed medical LLM 存在事实性不足、策略不合规和隐私披露薄弱问题。

#### 主要贡献

1. 提供大规模医疗 LLM hallucination 与部署风险评测。
2. 说明医疗场景不能只看回答流畅度，必须看安全性、合规性和高风险误答。
3. 释放了可支撑后续研究的结构化安全评测数据。

#### 和你论文的关系

这篇论文虽然不直接研究“意图识别”，但它能为你论文的动机和实验终点提供非常强的医疗场景支撑：

> 你做输入端治理，不只是为了让分类更准，而是为了减少医疗场景中的高风险 hallucination 与 unsafe direct answer。

也就是说，你的论文最终不能只停在：

- 意图分类 `micro-F1 / macro-F1`
- `Q_sem` 分段命中率

还应该尽量落到更贴近医疗风险的终点指标上。

#### 你可以学习的思路

- 将最终评测从“路由是否合理”推进到“是否真正减少高风险误答”。
- 在低质量测试集上增加医疗风险标签，例如：
  - 错误用药建议风险
  - 误导性诊断风险
  - 本应追问却直接答复

#### 可借鉴的实验方法

- 在你的测试集中加入“高风险医疗样本桶”。
- 分别比较：
  - 直接 RAG
  - `Q_sem` 路由
  - `Q_sem + ask/rewrite`
- 单独报告高风险桶上的错误率下降。

#### 可借鉴的数据集 / 实验设计 / 指标

- 你可以新增指标：
  - `unsafe medical answer rate`
  - `high-risk direct-answer rate`
  - `risk-bucket error reduction`
  - `conservative-trigger precision`

#### 可引用的地方

- 绪论：支撑医疗 LLM 部署中的 hallucination 风险是现实问题。
- 实验动机：说明需要高风险场景专项评测，而不是只看通用准确率。

## 对你当前硕士论文最直接的推进建议

今天这批文献对你最值得落实的，不是再扩标签，而是把实验结构改得更像一篇完整论文：

1. **把 route prediction 单独做成任务**
   - 不再只把 `Q_route` 当经验规则。
   - 先给样本标 `direct / rewrite / ask / conservative`。

2. **做一个轻量级 route baseline**
   - 延续你现有的 `TF-IDF + LR` 风格。
   - 至少比较 `rule-based`、`TF-IDF + LR`、`TF-IDF + SVM`。

3. **把 `ask` 从阈值触发升级为“是否值得追问”**
   - 增加 `need-ask` 与 `critical-slot hit` 标注。
   - 单独评估追问触发是否合理。

4. **把最终指标拉回医疗风险**
   - 不只看分类 F1。
   - 增加 `unsafe direct-answer rate`、`high-risk error rate`、`necessary-ask recall`。

## 今天最适合直接引用的句子级思路

你后面写论文时，可以把核心问题进一步收紧成下面这种表达：

> 医疗 RAG 的 hallucination 并不只来源于知识缺失，也来源于患者输入中的多意图纠缠、关键槽位缺失与可答性不足。  
> 因此，系统应在检索与生成前显式识别输入的意图完整性与不确定性，再决定直接回答、改写、追问或保守回复。

这个表述和你当前的多意图识别、`Q_sem`、`Q_route` 框架是对齐的，而且比“做一个意图分类器”更像完整硕士论文问题。
