## 总体判断

`2023 Q1` 这一桶依然没有直接命名为“医疗 RAG 幻觉治理”的代表性论文，但已经能看到一条更接近工程落地的路线：

1. **clarify before answer**：把澄清问题生成做成冷启动可用模块。
2. **intent before retrieval**：先解释上下文搜索意图，再决定如何改写或检索。
3. **OOD / unknown intent gating**：先识别超域、未知、越界输入，再决定是否保守处理。
4. **rewrite for noisy entities**：在检索前修复错实体、错别字、口语噪声。
5. **medical intent/entity parsing**：医疗查询理解开始出现更贴近真实场景的数据集与标注。

和你当前的方向相比，这一桶最重要的不是“大模型回答更强”，而是 **RAG 前输入控制器的各子模块开始分化清楚了**。

## 论文 1

### Zero-shot Clarifying Question Generation for Conversational Search

- 作者：Zhenduo Wang, Yuancheng Tu, Corby Rosset, Nick Craswell, Ming Wu, Qingyao Ai
- 首次公开时间：2023-01-30（arXiv v1）
- 链接：[arXiv](https://arxiv.org/abs/2301.12660)

#### 摘要

论文直接处理**歧义查询的意图识别与澄清问题生成**。作者指出，现实中缺少大规模真实 conversational search 澄清数据，导致 clarifying question 模型难以落地。为解决冷启动问题，论文研究 zero-shot clarifying question generation，并用问题模板与 query facets 约束生成结果，使问题更精确、更可用。

#### 主要贡献

1. 把 clarifying question generation 放到 **zero-shot / cold-start** 场景下讨论。
2. 提出受模板和 query facets 约束的澄清问题生成框架，降低生成发散。
3. 在零样本设定下超过已有 baseline，并用人工评估验证自然性与有用性提升。

#### 和你论文的关系

这篇最接近你的 `ask` 路由。你的医疗 RAG 现在最大问题之一是：患者问题是否缺关键槽位，是否应先追问。它给你的直接启发是：

- 追问模块不一定要依赖大规模人工多轮医疗对话数据，**可以先从 zero-shot / template-constrained 方式启动**。
- `Q_route` 可以显式输出 `need_ask`，再调用澄清问题模板或生成器。
- 医疗场景下可把缺失信息限制为少量高价值槽位，如 `部位 / 持续时间 / 程度 / 年龄 / 检查结果 / 用药史`。

#### 你可以学习的思路、实验方法、数据集、实验设计、评测指标、可引用点

- 思路：先判别歧义，再生成受约束的追问。
- 实验方法：比较 `直接回答`、`模板追问`、`zero-shot 追问`。
- 数据集：从你现有患者问题中抽取“信息缺一块就不宜检索”的样本，手工补 gold missing slots。
- 实验设计：同时评估 `need_ask` 识别和追问后下游检索收益。
- 评测指标：`ask precision/recall`、`slot completion rate`、`post-clarification Recall@k`、`unsafe direct answer rate`。
- 可引用点：澄清式交互可以在冷启动条件下先落地，不必等到完整多轮医疗语料齐备。

## 论文 2

### KG-ECO: Knowledge Graph Enhanced Entity Correction for Query Rewriting

- 作者：Jinglun Cai, Mingda Li, Ziyan Jiang, Eunah Cho, Zheng Chen, Yang Liu, Xing Fan, Chenlei Guo
- 首次公开时间：2023-02-21（arXiv v1）
- 链接：[arXiv](https://arxiv.org/abs/2302.10454)

#### 摘要

论文讨论 query rewriting 中一个很实际的问题：当用户输入里含有**错误实体**时，系统即使理解大意，也可能因为实体错写而误检索、误回答。作者提出 `KG-ECO`，把知识图谱的结构信息与文本描述引入实体纠错式 query rewriting，用于检测损坏实体片段、召回候选实体并重排序。

#### 主要贡献

1. 把 query rewriting 细化为**实体纠错**这一高风险子任务。
2. 同时建模 corrupt entity detection、entity retrieval 和 re-ranking。
3. 引入知识图谱结构与文本描述，在少样本和稀有实体场景下更有效。

#### 和你论文的关系

这篇对你的 `rewrite` 路由很重要。医疗输入里大量问题不是完全不清楚，而是：

- 药名、疾病名、检查名写错。
- 症状描述口语化、别名化。
- 把相近实体混写在一起。

这种情况下，如果系统直接进检索，很容易召回错误医学证据，后面的生成就会“在错检索上自信回答”。因此它能支撑你把 **输入侧实体噪声** 视作 hallucination 的上游诱因。

#### 你可以学习的思路、实验方法、数据集、实验设计、评测指标、可引用点

- 思路：把 rewrite 拆成 `去噪`、`纠错`、`补全` 三类，而不是统一重写。
- 实验方法：比较 `raw query`、`全文改写`、`实体纠错后检索`。
- 数据集：针对 MedDialog-CN 构造症状名、药名、科室名、疾病名的错拼/俗称/简称集。
- 实验设计：先测实体纠错，再测纠错后的检索与回答改善。
- 评测指标：`entity correction accuracy`、`Recall@k`、`evidence relevance`、`intent preservation rate`。
- 可引用点：对检索增强系统而言，输入中的实体错误是误检索和后续幻觉的重要前置来源。

## 论文 3

### Intent Identification and Entity Extraction for Healthcare Queries in Indic Languages

- 作者：Ankan Mullick, Ishani Mondal, Sourjyadip Ray, R Raghav, G Sai Chaitanya, Pawan Goyal
- 首次公开时间：2023-02-19（arXiv v1）
- 链接：[arXiv](https://arxiv.org/abs/2302.09685)

#### 摘要

论文针对医疗查询理解，构建了医疗查询意图与实体抽取数据集，包括 `IHQID-WebMD`、`IHQID-1mg` 以及真实医院查询数据，覆盖英语和多种 Indic 语言。目标是同时识别查询意图并抽取对应实体，从而支撑更实用的医疗 NLU 系统。

#### 主要贡献

1. 直接面向**医疗查询**构建意图识别 + 实体抽取数据资源。
2. 同时评估只用英语数据和使用目标语言数据两种更贴近现实成本约束的场景。
3. 强调医疗领域中的上下文相关实体与意图需要联合建模。

#### 和你论文的关系

这篇虽然不是为“hallucination 治理”提出的，但和你的硕士课题有很强的数据层关系：

- 你的系统也是先做输入理解，再决定是否进 RAG。
- 论文说明医疗查询本身就值得被拆成 `intent + entity` 两层，而不是一句话只给一个粗标签。
- 你的 `Q_med`、`Q_intent` 和后续 `rewrite / ask` 路由都可以建立在这种联合解析之上。

#### 你可以学习的思路、实验方法、数据集、实验设计、评测指标、可引用点

- 思路：医疗输入质量判断前，先做 `intent + entity` 联合解析。
- 实验方法：把你现有 12 类意图标签与症状/部位/时间/检查/药物实体结合起来。
- 数据集：参考其做法，为你自己的患者问题补小规模实体层标注。
- 实验设计：对比 `只有意图标签` 和 `意图+实体联合特征` 对 `Q_sem`、`Q_route` 的帮助。
- 评测指标：`intent micro/macro-F1`、`entity F1`、`route accuracy`、`ask trigger recall`。
- 可引用点：医疗查询理解应同时覆盖意图与医学实体，否则仅靠粗粒度意图难以支撑安全检索与回答。

## 论文 4

### A Hybrid Architecture for Out of Domain Intent Detection and Intent Discovery

- 作者：Masoud Akbari, Ali Mohades, M. Hassan Shirali-Shahreza
- 首次公开时间：2023-03-07（arXiv v1）
- 链接：[arXiv](https://arxiv.org/abs/2303.04134)

#### 摘要

论文关注任务型对话系统里非常关键但经常被低估的问题：输入可能是 `OOS/OOD`，即超出系统已知能力边界。作者提出一个混合架构，先用 VAE 区分已知意图和未知意图，再对 OOD/OOS 输入做无监督聚类，进一步发现潜在新意图并生成伪标签。

#### 主要贡献

1. 同时处理 **OOD intent detection** 和 **intent discovery**，而不是只做二分类拒识。
2. 用生成式表征学习提升已知/未知意图边界识别。
3. 在英语和波斯语数据上验证了检测与聚类发现的有效性。

#### 和你论文的关系

这篇和你的 `conservative` 路由几乎直接相连。医疗 RAG 并不只怕“答错”，也怕：

- 用户问题根本不在当前系统覆盖范围内。
- 输入里混有非医学诉求、违规诉求、模糊诉求。
- 一部分是已知医疗意图，另一部分是未知或越界意图。

这意味着你的前置控制器不应只做 12 类多标签分类，还应显式判断 **known / unknown / mixed-risk**。

#### 你可以学习的思路、实验方法、数据集、实验设计、评测指标、可引用点

- 思路：在意图分类器之外单独设一个 `unknown intent gate`。
- 实验方法：比较 `纯多标签分类` 与 `多标签 + OOD gate`。
- 数据集：补充 `非医学咨询`、`缺信息无法答复`、`含生活化/法律/用药决策越界内容` 等开放集样本。
- 实验设计：拆成 `已知意图`、`纯未知意图`、`已知+未知混合意图` 三类评测。
- 评测指标：`AUROC`、`FPR95`、`OOD recall`、`unsafe route rate`。
- 可引用点：可靠系统需要在回答前显式识别超域或未知意图，而不是把所有输入都强行映射到既有标签空间。

## 论文 5

### Large Language Models Know Your Contextual Search Intent: A Prompting Framework for Conversational Search

- 作者：Kelong Mao, Zhicheng Dou, Fengran Mo, Jiewen Hou, Haonan Chen, Hongjin Qian
- 首次公开时间：2023-03-12（arXiv v1）
- 链接：[arXiv](https://arxiv.org/abs/2303.06573)

#### 摘要

论文关注 conversational search 中的上下文意图理解。作者提出 `LLM4CS`，把大语言模型作为文本式搜索意图解释器，生成多个 query rewrites 和 hypothetical responses，再聚合成更鲁棒的上下文意图表示，用于提升检索。

#### 主要贡献

1. 把 LLM 用作**检索前的意图解释器**，而不是直接答案生成器。
2. 同时使用多路 rewrite 与 hypothetical response 来逼近真实上下文搜索意图。
3. 在多个 CAsT 基准上超过现有方法，并有人工评测支持。

#### 和你论文的关系

这篇和你的研究思路非常一致：LLM 的价值不一定在“直接回答患者”，而可能在 **回答前先解释患者意图、补足检索视角**。它能给你两点很强的论据：

- 你可以把 LLM 放在 RAG 之前，作为 `intent interpreter / rewrite suggester`。
- `rewrite` 不一定是单一结果，可以生成多个候选解释，再由检索得分或规则决定是否采用。

#### 你可以学习的思路、实验方法、数据集、实验设计、评测指标、可引用点

- 思路：让 LLM 先生成 `multiple rewrites / hypothetical intents`，再做路由或检索。
- 实验方法：比较 `单一路由 rewrite` 与 `多候选 rewrite + rerank`。
- 数据集：从患者问题中抽取有代词、省略、跨轮依赖、口语跳跃的问题做小规模评测集。
- 实验设计：同时测 rewrite 质量和下游检索收益，避免只看表面通顺度。
- 评测指标：`Recall@k`、`MRR`、`rewrite helpfulness`、`intent preservation rate`。
- 可引用点：大模型在检索增强系统中的关键作用之一，是先把上下文意图解释清楚，再决定如何检索。

## 论文 6

### From Baseline to Top Performer: A Reproducibility Study of Approaches at the TREC 2021 Conversational Assistance Track

- 作者：Weronika Lajewska, Krisztian Balog
- 首次公开时间：2023-01-25（arXiv v1）
- 链接：[arXiv](https://arxiv.org/abs/2301.10493)

#### 摘要

这篇不是新方法论文，而是对 TREC Conversational Assistance Track 系统的复现研究。它重新检查 baseline 和最佳系统的两阶段检索流水线，分析 query rewriter、fine-tuning 数据集和不同 pipeline 组件对整体效果的影响。

#### 主要贡献

1. 从复现角度拆解 conversational assistance pipeline，而不是只报最终分数。
2. 发现 query rewriting 的收益与**所处流水线阶段**、**所用训练数据** 强相关。
3. 给出可迁移的实践建议，强调 end-to-end 系统评估的重要性。

#### 和你论文的关系

这篇对你特别有用的不是模型本身，而是实验观：

- `rewrite` 是否有效，要放在完整 pipeline 中看，而不是只看文本改写本身。
- 你后面做 `direct / rewrite / ask / conservative` 路由时，也要避免只测单模块准确率。
- 医疗 RAG 里最值得抄的是它的 **流水线消融** 思路。

#### 你可以学习的思路、实验方法、数据集、实验设计、评测指标、可引用点

- 思路：把输入治理系统拆成多个组件逐个消融。
- 实验方法：分别关闭 `intent`、`rewrite`、`ask`、`conservative` 模块看整体变化。
- 数据集：可先在你现有 5 万条患者问题中抽样，再补一个低质量输入评测集。
- 实验设计：不要只报意图分类 F1，要报对最终检索与回答的净增益。
- 评测指标：`route accuracy`、`Recall@k`、`final answer safety`、`abstention utility`。
- 可引用点：输入治理模块的价值需要在完整检索流水线中验证，而不是孤立验证。

## 与硕士论文进展的关系

对照 `[[硕士论文进展]]`，今天这一桶最能强化你现在的论文主线：

1. **医疗输入不是天然可直接检索的**，需要先做意图与实体解析。
2. **不是所有低质量输入都该追问**，有些更适合 `rewrite / denoise / entity correction`。
3. **不是所有输入都该强行进入已知标签空间**，要有 `unknown / mixed-risk` 检测和 `conservative` 出口。
4. **LLM 的前置价值大于后置价值**：它可以先当意图解释器、改写器、澄清问题生成器。
5. **实验评估要往 end-to-end 走**：从意图识别分数，走向路由正确率、检索收益和最终回答安全性。

这和你当前的 `Q_sem` 可以进一步整合成下面这条更清晰的控制链：

```text
患者输入
-> intent + entity parsing
-> 信息充分性 / 医学相关性 / known-vs-unknown 判别
-> 路由决策：direct RAG / rewrite / ask / conservative
-> 检索
-> 生成
```

## 我可以直接学习和复用的内容

### 1. 可提炼成论文方法的思路

- 把 `Q_route` 从经验分数升级为**显式四路分类器**：`direct / rewrite / ask / conservative`。
- 在 `rewrite` 内部再区分：`minimal rewrite`、`entity correction`、`noise cleaning`。
- 在多标签意图分类器旁边增加 `unknown intent gate`，避免所有输入都被硬塞进 12 类。
- 让 LLM 主要负责前置解释：`候选 rewrite`、`候选澄清问题`、`缺失槽位识别`。

### 2. 可直接做的实验方法

- 做一个 `route-first` 实验：先路由，再检索回答；与 `all-direct-RAG` 比较。
- 做一个 `rewrite type` 消融：`无改写 / 完全重写 / 最小改写 / 实体纠错改写`。
- 做一个 `OOD gate` 消融：有无 `unknown intent detection` 对 unsafe answer rate 的影响。
- 做一个 `clarification utility` 实验：追问前后检索命中率和最终答案安全性变化。

### 3. 可补的数据集与标注

- 从现有 5 万条患者问题中补一个小规模 `route label` 集：`direct / rewrite / ask / conservative`。
- 再补一个 `missing slot` 集：部位、时长、程度、年龄、检查结果、用药史。
- 再补一个 `entity noise` 集：错别字、俗称、药名简称、疾病别名。
- 再补一个 `OOD / mixed-risk` 集：非医学、越界求诊断、需人工就医判断、信息极缺失。

### 4. 可重点关注的评测指标

- 多标签分类：`micro-F1`、`macro-F1`、`hamming loss`。
- 路由：`route accuracy`、`ask recall`、`conservative precision`。
- 检索：`Recall@k`、`MRR`、`nDCG@k`。
- 安全性：`unsafe direct answer rate`、`abstention utility`、`post-clarification gain`。
- 改写质量：`intent preservation rate`、`entity correction accuracy`、`rewrite helpfulness`。

### 5. 这一桶最值得引用的论述角度

- 幻觉治理不应只放在生成阶段，**输入歧义、错误实体、未知意图** 都是上游诱因。
- 医疗 RAG 更需要 `ask / rewrite / conservative` 这样的前置控制器，因为错误输入会放大为错误检索和错误医学回答。
- 大模型在医疗场景中未必首先用来“直接答”，更适合先做**意图解释、澄清触发与改写建议**。

## 今天这桶的结论

如果把 `2022 Q4` 看成你课题的几个前身模块开始汇合，那么 `2023 Q1` 更像是这些模块开始进入**可部署、可评测、可拆分消融**的阶段。对你最有价值的启发不是去追一个大而全的医疗生成系统，而是继续把论文收束为：

**面向医疗 RAG 的输入治理控制器：先识别意图、充分性与越界风险，再决定 direct / rewrite / ask / conservative 路由，以降低误检索和输入侧 hallucination。**
